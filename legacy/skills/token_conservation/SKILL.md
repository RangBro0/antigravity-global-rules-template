# 🪙 토큰 사용량 최소화 및 컨텍스트 다이어트 가이드라인 (Token Conservation Guide)

본 가이드라인은 에이전트가 사용자의 작업을 수행할 때 **컨텍스트 크기 증가를 예방하고 일일 토큰 비용을 최소화**하기 위해 반드시 준수해야 하는 행동 수칙입니다. 이 스킬은 대량의 코드 리포지토리 분석, 장기 디버깅 세션, 혹은 큰 파일 읽기/쓰기가 감지될 때 동적으로 로드되어 에이전트의 동작을 강제합니다.

---

## ⚠️ 시스템 도구 토큰 사용량 증가의 주요 원인

1. **대용량 소스 파일 전체 읽기**: 파일 크기가 큰 코드를 제한 없이 읽어 올 경우 컨텍스트가 급격하게 증가합니다.
2. **필터링 없는 장황한 명령어 출력**: 빌드, 테스트, 탐색 등의 명령어 실행 시 발생하는 다량의 로그를 그대로 컨텍스트에 담을 때 토큰이 과도하게 낭비됩니다.
3. **무분별한 웹 검색 및 원문 수집**: 웹 문서 원본을 가공 없이 텍스트 그대로 컨텍스트에 삽입할 때 발생합니다.
4. **누적된 대화 히스토리로 인한 캐시 미스**: 대화 턴이 누적되면 매 질문 시 이전 도구 호출 결과 및 히스토리가 매번 다시 입력 프롬프트로 들어가 대량의 토큰을 재소모합니다.

---

## 🔍 1. 코드 슬라이싱 (Code Slicing) - 파일 부분 읽기

에이전트는 100줄이 넘어가는 소스 코드를 통째로 읽어서는 안 되며, 필요한 부분만 정밀하게 타겟팅하여 슬라이싱해서 읽어야 합니다.

### [실행 방법]
1. **사전 검색 단계**:
   * 소스 파일의 전체 코드를 먼저 불러오지 말고, `grep_search` 도구를 사용하여 찾고자 하는 특정 함수명, 클래스명, 또는 에러 키워드를 검색합니다.
   * 검색 결과에서 해당 코드의 파일 경로와 정확한 **시작 라인 번호(Line Number)**를 확보합니다.
2. **부분 읽기 단계**:
   * `view_file` 도구를 호출할 때 `StartLine`과 `EndLine` 매개변수를 **반드시** 지정하여, 해당 부분 주변의 약 50~100줄만 잘라내어 읽습니다. (예: 시작 라인이 276라인인 경우 `StartLine: 270`, `EndLine: 320`으로 지정)
3. **전체 파일 가독 제한**:
   * 전체 소스 코드 크기가 500줄을 넘어가는 파일의 경우, 사용자가 명시적으로 파일 전체 출력을 요구하지 않는 한 `view_file`에서 라인 범위를 생략해서 호출하지 마십시오.

---

## 💻 2. 터미널 출력 및 정보 큐레이션 (Information Curating)

에이전트는 명령어 실행 결과나 검색 결과 등에서 흘러넘치는 불필요한 텍스트로 컨텍스트를 오염시켜서는 안 됩니다.

### [실행 방법]
1. **출력물 필터링**:
   * 빌드 명령어나 테스트 명령어를 구동할 때 발생 가능한 수만 줄의 성공 로그를 방지하기 위해, 에러 확인 목적이라면 오류 관련 라인만 필터링하거나 출력 길이를 제한하십시오.
   * 필요시 PowerShell/CMD의 `select`나 파일 리다이렉션 등을 활용하여 필요한 핵심 분석 로그만 콘솔에 뿌리도록 처리합니다.
2. **웹 검색 요약**:
   * `search_web` 또는 URL 읽기 도구를 활용할 때 원문 전체를 컨텍스트에 긁어오지 마십시오. 파싱 시 핵심 결론만 추려진 데이터만 컨텍스트에 삽입하도록 주의하십시오.

---

## 👥 3. 리서치 서브에이전트 위임 (Subagent Delegation)

광범위한 코드베이스 탐색, 파일 목록 스캔, 라이브러리 공식 문서 탐색 등의 '리서치 전용 태스크'는 메인 에이전트의 컨텍스트를 비대하게 만듭니다.

### [실행 방법]
1. **리서치 전용 subagent 기동**:
   * 많은 파일 읽기 및 다중 검색이 수반되는 기초 조사 단계에서는 `research` subagent를 스폰하여 태스크를 위임합니다.
   * `research` subagent가 수많은 도구 호출 결과와 문서를 읽고 나면, 메인 에이전트는 최종적으로 요약 보고된 핵심 내용만 보고받아 메인 컨텍스트의 과부하를 차단합니다.

---

## 🧹 4. 수동 컴팩션 및 세션 리셋 (Context Compaction)

컨텍스트의 크기가 지속적으로 누적되면 매 턴 질문마다 전체 대화 기록이 프롬프트로 재전송되어 기하급수적인 토큰 낭비를 유발합니다.

### [실행 방법]
1. **임계치 감시**:
   * 대화의 진행 단계(Turn)가 30회에 도달하거나, 현재 활성 컨텍스트 크기가 100,000 토큰을 초과하는 경우를 감지하십시오.
2. **사용자 권고 및 세션 전환**:
   * 임계치 도달 시 사용자에게 다음과 같이 권고하십시오:
     > *"현재 대화 세션이 길어져 매 질문마다 많은 토큰이 소모되고 있습니다. 지금까지 수행한 내용의 중간 요약본(Summary)을 제공해 드릴 테니, 새 대화창(New Conversation)을 열고 이 요약본을 붙여넣어 이어서 진행하는 것을 권장합니다."*
   * 승인 시 깔끔한 인계 요약본을 아티팩트로 작성하여 전달하고 대화를 리셋하도록 유도합니다.

---

## ⚡ 5. 컨텍스트 캐싱 (Context Caching) 활성화

자주 변경되지 않는 초기 시스템 세팅 정보, 거대한 기본 프롬프트 규격, 대규모 소스 코드 헤더 덩어리는 캐싱(Caching)을 염두에 두고 구조화합니다.

### [실행 방법]
1. **설정 동결**:
   * 초기 시스템 환경 변수 세팅이나 설정 파일 파싱 등은 대화의 가장 첫머리에 완료하고, 고정된 컨텍스트로 유지하여 캐시 히트율(Cache Hit Rate)을 극대화합니다.

---

## 💾 6. 경로 및 설정의 저장소 우선 탐색 (Repository-First Lookups)

에이전트는 깃허브 레포지토리 목록, 구글 드라이브 동기화 경로 등 환경 설정 정보를 파악할 때 로컬 파일 시스템이나 API를 직접 매번 스캔해서는 안 됩니다.

### [실행 방법]
1. **장기 메모리 및 데이터베이스 우선 조회**:
   * 동기화 경로, 레포지토리 주소, 작업 환경 변수 등의 핵심 메타데이터는 MEM0 그래프(`read_graph`)나 설정 데이터베이스(SQLite `sync_configs` 테이블)를 가장 먼저 조회하여 획득하십시오.
2. **검색 최소화**:
   * MEM0 또는 DB에 설정 정보가 존재한다면 추가적인 파일 검색(`Get-ChildItem` 등)이나 깃허브 API 탐색(`search_repositories` 등) 단계를 완전히 생략하여 토큰 및 자원 소모를 최소화합니다.
