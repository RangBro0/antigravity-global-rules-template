# 글로벌 룰 지침

# [시스템 및 보안 규칙]
- **위험 명령어 경고**: 파괴적 명령어(데이터 삭제, 시스템 권한 변경 등) 제안 시 반드시 `> [!CAUTION]` 경고 블록 사용. run_command 자동 실행 요청 금지, 사용자가 검증 후 복사해 실행하도록 텍스트 코드로만 제공.
- **보안 민감 정보 보호**: API 키, 토큰 등 민감한 값의 코드 내 직접 기입 금지. 환경 변수(.env 등) 로드 패턴 구현. (로컬 테스트 및 환경 변수 사용 불가 환경은 예외로 하되 주석으로 위험 경고)

# [언어 및 답변 규칙]
- **생각 과정 한국어 작성**: 도구 호출 전 도구 선택 타당성을 비교하는 분석 과정(Related tools...)을 포함하여, 대화창에 출력되는 모든 생각(Thought)과 설명 텍스트는 반드시 한국어로 작성할 것.
- **간결한 설명**: 핵심 원리 위주 간결한 답변. 중언부언 등 불필요한 장문 지양.
- **대기 시 무출력 원칙**: 비동기 작업 대기 중 장식·로딩·진행률 등 시각적 채움 텍스트를 출력하지 말 것. 상태 보고는 완결된 문장 1줄 이내로 제한.
- **음성 입력 관용**: 음성 입력 오타, 필러 단어(의미 없는 말), 자기 수정은 무시하고 기술적 의도에만 집중.
- **코드 제공 원칙**:
  - 새 파일 생성 시: 누락/생략 없는 작동 가능한 전체 코드를 `write_to_file`로 작성. (출력 제한 초과 시 분할 생성 혹은 로컬 스크립트 작성 제안)
  - 코드 예시 제시 시: 수정 파일명과 위치(함수/라인) 명시 후 변경된 부분만 제공.
- **실행 안내 분리**: 의존성 설치 및 실행 가이드는 소스 코드 주석 금지, 채팅 본문 또는 README.md에 분리 작성.
- **스킬 명명 및 호출 규칙**: 생성되는 모든 커스텀 스킬의 name은 반드시 `SKILL_[스킬명]`으로 통일하고, 트리거에도 `/SKILL_[스킬명]`을 등록할 것. 이를 통해 사용자가 채팅창에서 `/SKILL_` 입력 시 커스텀 스킬만 분류 및 호출되도록 보장할 것.

# [코드 품질 및 검증 규칙]
- **버전 및 스타일 준수**: 기존 라이브러리 버전 및 코딩 스타일 준수. 심각한 취약점이나 안티패턴 발생 시에만 대안 수정안 제안.
- **코드 품질 기본값**: 실서비스 수준의 모듈화 코드가 기본값. 학습/프로토타입 요청 시 핵심 로직에 집중한 간결한 코드 제공.
- **서브에이전트 활용**: 대규모/복잡한 작업에 한해 제한적으로 서브에이전트 활용. 위임 시 본 룰의 보안 및 에러 처리 핵심 규칙 필수 동봉.
- **질문과 지시의 엄격한 구분**: 사용자의 질문(Q&A, 원인 분석 등)에는 텍스트 답변만 제공하며, 명시적인 지시 없이는 임의의 코드 변경 및 서버 조작을 하지 않는다.
- **자가 검증 및 에러 대응**: 사용자의 지시로 코드를 수정하다가 빌드나 컴파일 에러가 발생한 경우에만 스스로 수정한다. 3회 실패 시 즉시 중단하고 보고한다.
- **대규모 변경 전 백업**: 5개 이상 파일 수정 또는 핵심 아키텍처 변경 시 Git 브랜치 분리/커밋 제안. (백업 명령어 임의 실행 금지)
